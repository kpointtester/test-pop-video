Feature 1: Tag-based Embed Loading (player-silk.js)

How it works: Embed div carries data-kvideo-tags instead of data-kvideo-id. The embed script fires a search XHR (/api/v3/search?type=videos&facet.tag=...), then client-side exact-matches the video tag set against the embed tag set (sorted, case-folded). On match it injects the player iframe; on no-match the container stays blank.

Feature 2: Bandwidth-based Resolution (Init and Runtime)

How it works: data-video-params='{"bandwidth":"N"}' on the embed div. The embed script converts the bandwidth value to a res=N param appended to the iframe URL. Inside kplayer.min.js, KPlayer.loadMedia() maps res to an HLS.js level index and sets this.hls.nextLevel before playback starts. Runtime: kPlayer1.setBandwidth(N) posts a message to the iframe; the inner player calls o.hlsPlayer.autoLevelCapping = n and conditionally o.hlsPlayer.nextLevel = n.

Root cause — setBandwidth only forces DOWN, not UP

Source of the inner player.setBandwidth (decompiled from kplayer.min.js): 
function(e) {
  if (o && "native_hls" === o.rendererName && o.hlsPlayer && o.hlsPlayer.levels && o.hlsPlayer.levels.length) {
    for (var t = 1000*e, n = 0, r = -1, i = 0; i < o.hlsPlayer.levels.length; i++)
      o.hlsPlayer.levels[i].bitrate <= t && o.hlsPlayer.levels[i].bitrate > r && (r = o.hlsPlayer.levels[i].bitrate, n = i);
    o.hlsPlayer.autoLevelCapping = n;            // sets the cap
    o.hlsPlayer.currentLevel > n && (o.hlsPlayer.nextLevel = n);  // forces DOWN only
  }
}
Explanation: When setBandwidth(8000) is called while playing at 240p, the condition currentLevel > n evaluates to false (240p index 0 is not > highest index 3), so nextLevel is never set. Only autoLevelCapping is updated. HLS.js ABR is theoretically free to climb, but in practice the bandwidth estimator stays conservative after a long period of 240p downloads and does not auto-upgrade within a reasonable timeframe.

Recommendation: After removing the cap (when new bw > current cap), also set o.hlsPlayer.nextLevel = -1 (full ABR mode) to prompt HLS.js to reassess quality immediately.

Known Issues

Bug 1 — kplayer.min.js: Cannot set properties of null (setting ‘nextLevel’)

Uncaught TypeError: Cannot set properties of null (setting 'nextLevel')
KPlayer.loadMedia @ kplayer.min.js:1

Trigger: Occurs in loadMedia() when this.hls is null at the moment the init-level is being set.
Observed in: CDP/automated context (Run 2). Does not reliably reproduce in real-browser user-gesture flow (Run 1).
Impact: player.hls stays null after the crash. Playback may still proceed via the o.hlsPlayer closure reference but runtime bandwidth control is degraded (LOW→HIGH does not work).
Recommendation: Add a null-guard in loadMedia() before setting this.hls.nextLevel: 
if (this.hls) { this.hls.nextLevel = targetLevel; }
Bug 2 — setBandwidth LOW→HIGH does not force quality rise

Trigger: Calling kPlayer1.setBandwidth(highValue) after quality has been capped to a low level.
Impact: ABR cap is lifted (autoLevelCapping updated) but HLS.js does not immediately reassess quality. In a fast-network environment quality may never rise within a playback session because the bandwidth estimator stays conservative.
Recommendation: In the inner setBandwidth implementation, when the new cap is higher than the current level, additionally set o.hlsPlayer.nextLevel = -1 to re-enable full ABR: 
o.hlsPlayer.autoLevelCapping = n;
if (o.hlsPlayer.currentLevel > n) {
  o.hlsPlayer.nextLevel = n;       // cap enforcement (existing)
} else {
  o.hlsPlayer.nextLevel = -1;      // lift cap → let ABR climb (NEW)
}

Automation Notes (CDP context)

  • Autoplay: Even with muted:true in data-video-params, Chrome CDP sessions may block autoplay after a page reload. Workaround: call video.play() directly inside the iframe via Selenium switch_to.frame() + execute_script().
  • Cross-origin iframe inspection: Selenium switch_to.frame() via CDP bypasses the same-origin policy, allowing direct JS execution inside the vega-automation iframe even when the parent page is on GitHub Pages.
  • kPlayer1 bridge: getPlaybackQuality() returns 0 until the video is actually playing. The postMessage handshake establishes only after first play start; window.location.reload() may not re-establish it — use window.location.href = url for fresh navigation instead.
  • HLS segment inspection: Use performance.getEntriesByType("resource") inside the iframe to verify which hls-Xp/playlist.m3u8 and seg-N.ts files were fetched.
